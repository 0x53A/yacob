# canopen-rs

CANopen protocol stack for embedded Rust. `no_std`, `no_alloc`, uses `embedded-can` 0.4 traits.

## Workspace

- `canopen-core/` — protocol implementation (NMT, SDO, PDO, heartbeat, EMCY, SYNC, LSS)
- `canopen-derive/` — proc macros (`object_dictionary!`, `object_dictionary_from_eds!`, `sdo_client_from_eds!`, EDS export)
- `canopen-linux/` — socketcan, SLCAN, and UDP multicast transports + SDO client helpers
- `hil-tests/` — hardware-in-the-loop test harness
- `interop-tests/` — python-canopen interop tests over UDP multicast (no root needed)
- `examples/stm32-node/` — Embassy firmware for Nucleo-G431KB (excluded from workspace, cross-compiled)
- `examples/stm32-fwupdate/` — Embassy firmware with CiA 302 firmware update over CANopen

## Build & Test

```sh
cargo build                  # workspace (Linux side)
cargo test                   # 168 unit + integration tests

# Interop tests (python-canopen over UDP multicast, no root needed)
cd interop-tests
uv run pytest -v             # 27 tests

cd examples/stm32-node
cargo build --release
probe-rs run --chip STM32G431KBTx target/thumbv7em-none-eabihf/release/stm32-canopen-node

# HIL tests (requires CAN hardware or vcan)
CAN_IFACE=can0 cargo test -p hil-tests -- --test-threads=1 --ignored
```

## Architecture

- All CAN communication goes through `embedded_can::nb::Can` trait (no custom Transport trait).
- `CanFrame` implements `embedded_can::Frame` for interop with any CAN driver.
- `MailboxTransport` is `Send + Sync` (uses `critical_section`) — safe for ISR-to-task bridging.
- `ObjectDictionary` trait: generated by proc macro, maps index/subindex to typed struct fields.
- `Node<OD>` ties together NMT, SDO server, PDO engines, heartbeat, EMCY producer, LSS slave. Call `node.process(&mut transport, &clock)` periodically.
- `NodeConfig::from_od(&od, node_id)` pulls PDO configs from the OD (via the `PdoConfigSource` trait the macro implements); override the rest with struct update syntax. `node.tpdo_cob_id(n)` / `rpdo_cob_id(n)` return resolved COB-IDs.
- `node.next_change()` drains OD events decoded into the generated `${Name}Change` enum (one variant per writable entry, carrying the current value) — no raw index/subindex matching. Backed by the `OdChanges` trait; `node.next_event()` still gives raw `OdEvent`s. **Transitional**: the event queue + drain APIs are slated for replacement by the three application models (inline / reactive / scan) — design agreed in `_Tasks/application-models.md`, not yet implemented.
- `SharedNode<OD, TPDO, RPDO>` (embassy feature) wraps the `static Mutex<RefCell<Option<Node>>>` pattern: `NODE.init(node)` once, then `NODE.with(|node| ...)`.
- `TransmissionType` enum ↔ raw CiA 301 `u8` via `.raw()` / `from_raw()`.
- `SdoClient<const BUF: usize = 256>` is the low-level poll-based state machine for SDO transfers. BUF controls max segmented transfer size (up to 889 for full CiA 301 block).
- `SdoDriver` is the async high-level SDO driver (borrows `AsyncCan`, drives full transfers). Has `upload_timed`/`download_timed` variants that accept a `Clock` and timeout in microseconds.
- `NmtCommand::to_frame(target)` builds NMT CAN frames (available in no_std for embedded masters).
- `NmtState::from_heartbeat_byte(b)` decodes heartbeat state bytes.
- `Node::request_reset(ResetType)` triggers a CANopen-layer reset from application code.
- `Node::events_dropped()` tracks event queue overflow count for debugging.
- `ObjectDictionary::validate_write()` pre-write hook called by SDO server before committing writes.
- `CanDemux` owns a transport and routes frames to per-protocol buffers (SDO, PDO, heartbeat). `DemuxSdoPort` implements `AsyncCan` so `SdoDriver` works seamlessly while non-SDO traffic is buffered.
- `SdoPort` is a simpler single-consumer filter with overflow buffer.
- SDO server supports expedited, segmented, and block transfers (upload + download) with CRC.
- All types use static allocation (heapless).
- `Dcf` (behind `alloc` feature) parses DCF/EDS files at runtime and applies configured values to local ODs or remote nodes via SDO.
- STM32 firmware uses Embassy channels to bridge async CAN driver and polling Node.

## OD Definition

Inline DSL (supports VAR, RECORD, ARRAY, and variable-length types):
```rust
object_dictionary! {
    #[export_eds(path = "./my_device.eds")]
    pub struct MyOd {
        [0x1000] device_type: u32 = 0x191, ro;
        [0x1008] device_name: visible_string<32>, ro;
        [0x1003] error_field: array<u32, 8>, ro;
        [0x6000] inputs: record {
            [1] input1: u8 = 0, ro, pdo;
        };
        [0x2020] firmware: domain<1024>, rw;

        tpdo[1](transmission_type = event_driven, inhibit_time = 50ms, event_timer = 1s) {
            input1,
        };
    }
}
```
PDO parameters accept CiA 301 keywords (`event_driven`, `event_driven_manufacturer`, `sync_acyclic`, `sync_cyclic(N)`, `rtr_sync`, `rtr_event`) or raw spec values (`transmission_type = 255`). Time parameters accept raw integers in spec wire units (inhibit_time in 100µs, event_timer in ms) or unit-suffixed literals: `50ms`, `500us`, `0.1s` (floats need a suffix; values must be exactly representable in the wire unit).

Generated besides the struct itself:
- `MyOd::EDS: &'static str` (uncompressed) and `MyOd::EDS_COMPRESSED: &'static [u8]` (deflate). Auto-registers 0x1021 (Store EDS, compressed) and 0x1022 (Store Format = 1) for device self-description via SDO.
- Address consts: `MyOd::INPUT1: (u16, u8)` per entry (usable as match patterns), `MyOd::ERROR_FIELD_INDEX: u16` for arrays.
- `MyOdChange` enum + `OdChanges` impl: one variant per writable entry (`Input1(u8)`, arrays as `Name(subindex, value)`, variable-length as unit variants), decoded by `node.next_change()`.
- `MyOdNode` type alias = `Node<MyOd, TPDO_COUNT, RPDO_COUNT>`.
- `PdoConfigSource` impl for `NodeConfig::from_od`.

Variable-length types: `visible_string<N>`, `octet_string<N>`, `domain<N>` backed by heapless. Use `#[use_alloc]` attribute for `String`/`Vec<u8>` instead (no capacity needed).

Write validation via `#[validate_write(fn_name)]` — the SDO server calls the named method before committing writes:
```rust
object_dictionary! {
    #[validate_write(check_value)]
    pub struct MyOd {
        [0x2000] level: u16 = 0, rw;
    }
}
impl MyOd {
    fn check_value(&self, index: u16, sub: u8, data: &[u8]) -> Result<(), OdError> {
        // Return Err(OdError::ValueRange) to abort the SDO write
        Ok(())
    }
}
```

From EDS file (with per-entry capacity overrides and automatic PDO extraction):
```rust
object_dictionary_from_eds! {
    pub struct MyOd = "device.eds"
    with {
        [0x1008] capacity = 32,
        [0x1021] capacity = 4096,
    };
}
```

Typed async SDO client from EDS:
```rust
sdo_client_from_eds! {
    pub struct DeviceClient = "device.eds";
}
let client = DeviceClient::new(node_id);
let status: u16 = client.read_statusword(&mut can).await?;
client.write_controlword(0x0F, &mut can).await?;
```

Runtime DCF configuration (requires `alloc` feature):
```rust
let dcf = canopen_core::dcf::Dcf::parse(dcf_text)?;
dcf.apply(&mut od);                               // local OD
dcf.apply_remote(&sdo_driver, &mut can).await?;    // remote node via SDO
```

## Async SDO

`AsyncCan` trait for async CAN transports. `SdoDriver` drives complete SDO transfers:
```rust
let sdo = SdoDriver::new(target_node);
let mut demux = CanDemux::new(transport);
let mut port = demux.sdo_port(target_node);
let val = sdo.read_u32(0x1000, 0, &mut port).await?;
// Non-SDO frames buffered in demux:
while let Some(hb) = demux.try_recv_heartbeat() { ... }
```

## Features (canopen-core)

| Feature | Default | Effect |
|---|---|---|
| `alloc` | off | Enables `Dcf` runtime parser, `extern crate alloc` |
| `std` | off | Enables std error impls, implies `alloc` |
| `embassy` | off | Enables `OdEventSignal` (embassy_sync) and a real timer for `SdoDriver` timeouts on silent buses (embassy_time) |
| `defmt` | off | Enables defmt formatting |

## Transports

| Transport | Crate | Needs root | Use case |
|---|---|---|---|
| `SocketcanTransport` | canopen-linux | Yes (vcan setup) | Linux socketcan |
| `SlcanTransport<P: SerialPort>` | canopen-core | No | Serial SLCAN adapters |
| `UdpMulticastTransport` | canopen-linux | No | Cross-process testing, CI |
| `MailboxTransport` | canopen-core | N/A | ISR↔task bridging on MCU |

All implement `embedded_can::nb::Can<Frame = CanFrame>`.

The SLCAN protocol driver (`canopen_core::slcan`) is sans-IO and generic over
the `SerialPort` byte-stream trait, so it works in `no_std` (e.g. over an MCU
UART). Backends: `canopen_linux::slcan::UnixSerialPort` (unix-only, with
`slcan::open(path, bitrate)` handling the CDC-ACM DTR-reset init dance) and
`canopen_core::slcan::IoPort<T>` (`std` feature) wrapping any
`std::io::Read + Write`, e.g. the `serialport` crate on Windows/macOS.

## HIL Test Setup

- Nucleo-G431KB connected via ST-Link (probe-rs)
- ESP32 SLCAN adapter on USB (creates socketcan interface via slcand)
- Both on the same CAN bus at 500 kbps
- SLCAN setup:
  ```sh
  sudo slcand -o -c -s6 /dev/ttyUSB0 slcan0   # s6 = 500kbps
  sudo ip link set up slcan0
  ```
- Run HIL tests: `CAN_IFACE=slcan0 cargo test -p hil-tests -- --test-threads=1 --ignored`

## STM32 Notes

- Nucleo-G431KB: STM32G431KBTx
- embassy-stm32 v0.4 with `stm32g431kb` feature
- FDCAN1 peripheral in classic CAN mode, 500 kbps
- Pins: PA11 (RX) / PA12 (TX)
