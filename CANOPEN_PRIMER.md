# CANopen Primer

CANopen is a higher-layer protocol built on top of CAN (Controller Area Network). It's standardised by [CiA (CAN in Automation)](https://www.can-cia.org/) and widely used in industrial automation, medical devices, and embedded systems.

## Core Concepts

**Object Dictionary (OD)** — the central data model of every CANopen device. A table of entries addressed by 16-bit index + 8-bit subindex, holding configuration, status, and application data. Standardised index ranges:

- `0x1000–0x1FFF` — Communication profile (device type, heartbeat, SDO/PDO params)
- `0x2000–0x5FFF` — Manufacturer-specific
- `0x6000–0x9FFF` — Standardised device profile (CiA 401 for I/O, CiA 402 for drives, etc.)

**Node-ID** — each device on the bus has a unique ID (1–127).

## Communication Protocols

| Protocol | Purpose | CAN IDs (predefined) |
|----------|---------|---------------------|
| **NMT** | Network management (start/stop/reset nodes) | `0x000` |
| **Heartbeat** | Node alive monitoring | `0x700 + node_id` |
| **EMCY** | Emergency error messages | `0x080 + node_id` |
| **SDO** | Read/write arbitrary OD entries (request/response) | `0x600/0x580 + node_id` |
| **PDO** | Real-time process data (mapped OD entries, no overhead) | `0x180–0x500 + node_id` |
| **SYNC** | Synchronisation trigger for synchronous PDOs | `0x080` |
| **TIME** | Network time distribution | `0x100` |

**SDO (Service Data Object)** — client/server protocol for accessing any OD entry. Supports expedited (up to 4 bytes in one frame) and segmented transfers. Used for configuration, not real-time data.

**PDO (Process Data Object)** — efficient data exchange without protocol overhead. Each PDO maps specific OD entries into a CAN frame. Up to 4 TPDOs (transmit) and 4 RPDOs (receive) per node in the predefined connection set. Trigger modes: event-driven, timer-based, or SYNC-synchronous.

**NMT (Network Management)** — state machine: Initialisation → Pre-operational → Operational → Stopped. Only Operational state enables PDO communication.

For protocol details, see the [CiA 301 specification](https://www.can-cia.org/groups/specifications/) (the core CANopen application layer standard).

## File Formats

### EDS — Electronic Data Sheet (CiA 306)

Describes a **device type**. INI-style file listing all OD entries with data types, access rights, and default values. Not tied to a specific node or network — it's the generic template for a device model.

```ini
[1000]
ParameterName=Device Type
ObjectType=0x7
DataType=0x0007
AccessType=ro
DefaultValue=0x000000C8
PDOMapping=0
```

Tools: most CANopen configuration tools import EDS files.
Spec: [CiA 306 — Electronic Data Sheet](https://www.can-cia.org/groups/specifications/)

### DCF — Device Configuration File (CiA 306)

Same format as EDS, but describes a **configured device instance**. Contains concrete parameter values, node-ID, and baud rate for a specific device on a specific network. Typically generated by a configuration tool from an EDS.

### XDD / XDC — XML Device Description (CiA 311)

XML-based successors to EDS/DCF. XDD replaces EDS (device type), XDC replaces DCF (configured instance). Used by CANopen FD and shared with Powerlink.

Spec: [CiA 311 — XML Device Description](https://www.can-cia.org/groups/specifications/)

### DBC — CAN Database (Vector)

**Not a CANopen format.** Proprietary Vector CANdb format for raw CAN signal definitions (message IDs, signal bit positions, scaling, units). The de facto standard in automotive CAN tooling (CANalyzer, CANoe, etc.).

DBC operates at the CAN layer, not the CANopen layer. However, you can generate a DBC from a CANopen PDO configuration to let raw-CAN tools interpret PDO data without CANopen awareness. This only works for PDOs (fixed-layout data frames), not for SDO transfers or NMT commands.
