# canopen-rs

CANopen protocol stack for embedded Rust. `no_std`, `no_alloc`, uses `embedded-can` 0.4 traits.

## Workspace

- `canopen-core/` — protocol implementation (NMT, SDO, PDO, heartbeat, EMCY, SYNC, LSS)
- `canopen-derive/` — proc macros (`object_dictionary!`, `object_dictionary_from_eds!`, `sdo_client_from_eds!`, EDS export)
- `canopen-linux/` — socketcan, SLCAN, UDP multicast, and canwsd WebSocket transports + SDO client helpers
- `canwsd-proto/` — shared canwsd interface definition (no_std): WS wire format, filter syntax, REST paths, JSON command/status types, WS close codes. Implemented by `tools/canwsd` (Linux server), consumed by `canopen-web`/`canopen-linux` clients and embedded servers
- `tools/canwsd/` — Linux canwsd server (`expose`/`attach`/`list`); socketCAN over WebSocket. See `tools/canwsd/CLAUDE.md`
- `tools/canwsd-dump/` — a minimal `candump` for canwsd networks: listens to one remote network over WebSocket and prints frames (no vcan, no root). See `tools/canwsd-dump/CLAUDE.md`
- `hil-tests/` — hardware-in-the-loop test harness
- `interop-tests/` — python-canopen interop tests over UDP multicast (no root needed)
- `examples/stm32-node/` — Embassy firmware for Nucleo-G431KB (excluded from workspace, cross-compiled)
- `examples/stm32-fwupdate/` — Embassy firmware with CiA 302 firmware update over CANopen

## Build & Test

```sh
cargo build                  # workspace (Linux side)
cargo test                   # unit + integration tests

# Interop tests (python-canopen over UDP multicast, no root needed)
cd interop-tests
uv run pytest -v

# STM32 firmware (separate project)
cd examples/stm32-node
cargo build --release
probe-rs run --chip STM32G431KBTx target/thumbv7em-none-eabihf/release/stm32-canopen-node

# HIL tests (requires CAN hardware or vcan)
CAN_IFACE=can0 cargo test -p hil-tests -- --test-threads=1 --ignored
```

## Architecture

- All CAN communication goes through `embedded_can::nb::Can` trait (no custom Transport trait).
- `CanFrame` implements `embedded_can::Frame` for interop with any CAN driver.
- `MailboxTransport` is `Send + Sync` (uses `critical_section`) — safe for ISR-to-task bridging.
- `ObjectDictionary` trait: generated by proc macro, maps index/subindex to typed struct fields.
- `Node<OD>` ties together NMT, SDO server, PDO engines, heartbeat, EMCY producer, LSS slave. Call `node.process(&mut transport, &clock)` periodically.
- `NodeConfig::from_od(&od, node_id)` pulls PDO configs from the OD (via the `PdoConfigSource` trait the macro implements); override the rest with struct update syntax.
- **Public PDO APIs take CANopen PDO numbers, never dense engine slots**: `PdoNumber` (1..=512; `PdoNumber::of::<N>()` is compile-time checked, `PdoNumber::new`/`TryFrom<u16>` for runtime values). With sparse declarations (`rpdo[1]` + `rpdo[79]`), RPDO79 is addressed as 79 everywhere — `node.tpdo_cob_id(n)`/`rpdo_cob_id(n)` (return `Option<CobId>`), `rpdo_deadline_expired(n)`, and `TpdoEngine`/`RpdoEngine::config`/`config_mut`. Undeclared numbers answer `None`/`false`. Dense slots exist only crate-internally (`config_slot*`).
- `node.next_change()` drains OD events decoded into the generated `${Name}Change` enum (one variant per writable entry, carrying the current value) — no raw index/subindex matching. Backed by the `OdChanges` trait; `node.next_event()` still gives raw `OdEvent`s. **Transitional**: the event queue + drain APIs are slated for replacement by the three application models (inline / reactive / scan) — design agreed in `_Tasks/application-models.md`, not yet implemented.
- `SharedNode<OD, TPDO, RPDO>` (embassy feature) wraps the `static Mutex<RefCell<Option<Node>>>` pattern: `NODE.init(node)` once, then `NODE.with(|node| ...)`.
- `TransmissionType` enum ↔ raw CiA 301 `u8` via `.raw()` / `from_raw()`.
- `SdoClient<const BUF: usize = 256>` is the low-level poll-based state machine for SDO transfers. BUF controls max segmented transfer size (up to 889 for full CiA 301 block).
- `SdoDriver` is the async high-level SDO driver (borrows `AsyncCan`, drives full transfers). Has `upload_timed`/`download_timed` variants that accept a `Clock` and timeout in microseconds.
- `NmtCommand::to_frame(target)` builds NMT CAN frames (available in no_std for embedded masters).
- `NmtState::from_heartbeat_byte(b)` decodes heartbeat state bytes.
- `Node::request_reset(ResetType)` triggers a CANopen-layer reset from application code.
- `Node::events_dropped()` tracks event queue overflow count for debugging.
- `ObjectDictionary::validate_write()` pre-write hook called by SDO server before committing writes.
- **RPDO deadline monitoring** (CiA 301 event timer, RPDO comm param sub 5): DSL `rpdo[1](deadline = 500ms)` or SDO write to `0x140N` sub 5 (in Pre-Operational; PDO comm params are locked while Operational). Two surfaces with different semantics: the **level query** `node.rpdo_deadline_expired(PdoNumber)` reads "no in-deadline data" — true initially (before the first frame ever arrives, incl. right after entering Operational) and again once `elapsed > deadline`, cleared by reception; the **event channel** (`OdEventSource::RpdoDeadline`, index = comm param index, sub 0) arms on first reception and fires once per silence period — never before the counterpart has spoken. Mapped OD entries keep their last received values. Leaving Operational discards edge state (flag returns to "not fresh", no events); detection latency = `process()` period. Deadlines need margin over the sender's period (1.5–2×, like heartbeat consumer time). The stack never sends EMCY itself — apps report `EmcyErrorCode::RpdoTimeout` (0x8250) via `set_error`; `vcan_node.rs` shows the pattern (alarm on event, clear on flag).
- **Heartbeat OD integration**: 0x1017 (producer heartbeat time) is live-wired — when the OD declares it, its value wins over `NodeConfig::heartbeat_interval_ms` at init, SDO writes change the runtime period, 0 disables production. 0x1016 (consumer) entries `(node_id << 16) | timeout_ms` become up to `MAX_HEARTBEAT_MONITORS` (8) `HeartbeatMonitor`s (slot = subindex − 1; monitors whose entry is unchanged keep runtime state across SDO rewrites/resets). Monitoring starts with the first heartbeat or boot-up frame from the producer (CiA 301 / CANopenNode behavior) — a configured-but-silent node stays `Waiting` and never times out. Typed events via `node.next_heartbeat_event()`: `HeartbeatEvent::{Started, StateChanged, RemoteReset, Timeout, Recovered}` (own queue, not `OdEvent`); level query `node.heartbeat_status(node_id)` → `HeartbeatMonitorState::{Disabled, Waiting, Alive, Timeout}` with `.is_alive()` as the freshness flag (analogous to RPDO deadlines). Consumer timeouts are **not** gated on Operational (error control runs in every post-boot state). The SDO server validates 0x1016 downloads: invalid node id or reserved bits 24..32 → abort 0x0609_0030, same producer in two active entries → 0x0604_0043. No auto-EMCY (kept deliberately — `HeartbeatEvent::Timeout` is where an auto-trigger would make sense if ever added): apps report `EmcyErrorCode::HeartbeatError` (0x8130) themselves; `vcan_node.rs` shows the pattern.
- **Master application model (std)**: `bus::SharedCanBus` broadcasts every received frame to independent `Subscription`s (own bounded queue, optional publish-side `Fn(&CanFrame) -> bool` filter, per-subscription `OverflowPolicy` + overflow counter). `TxQueue` is a separate bounded non-lossy transmit queue; `run_pump` drives a `nb::Can` transport (RX → publish, TX queue → transmit) until stop or transport error — no lifecycle handling, tear down and restart on adapter loss. `SubscriptionPort` implements `AsyncCan` over `(TxQueue, Subscription)` so `SdoDriver`/EDS clients run on the bus; it is waker-backed (`recv_async`/`send_async` suspend until publish/`try_pop` wakes them — no spinning on idle, safe on tokio; multiple waiters supported, no fairness guarantee). Design + decisions: `_Tasks/master-application-model.md`.
- `events::CanOpenEvent::decode(&CanFrame)` (no_std) — semantic decode layer (NMT/SYNC/EMCY/heartbeat/PDO/SDO); assumes the pre-defined connection set.
- `CanDemux` owns a transport and routes frames to per-protocol buffers (SDO, PDO, heartbeat). `DemuxSdoPort` implements `AsyncCan` so `SdoDriver` works seamlessly while non-SDO traffic is buffered. **Legacy** for master-side code — superseded by the bus model; do not grow it.
- `SdoPort` is a simpler single-consumer filter with overflow buffer.
- SDO server supports expedited, segmented, and block transfers (upload + download) with CRC.
- **Additional SDO servers** (CiA 301 0x1201–0x127F): DSL `sdo_server[N](cob_rx = ..., cob_tx = ...)` declares extra channels alongside the implicit default (0x1200, `0x600/0x580 + node_id`) — e.g. one channel for normal ops, one for a diagnostics tool. Each is an independent `SdoServer` transfer state machine over the shared OD, so a transfer on one channel never disturbs another. COB-IDs are fixed (absolute or `node_id + base`, resolved at `Node::new`) and **const** — the 0x1201+ records are read-only, never remappable at runtime (the CiA 302 SDO Manager is the only spec reason to remap and is out of scope). Threaded through a `const SDO: usize = 0` generic on `Node`/`NodeConfig` (default 0 = zero cost); the macro emits `SDO_COUNT`, `SdoServerConfigSource` (pulled by `NodeConfig::from_od`), and the OD mirror-back (`store_sdo_server_cob_ids`, an `ObjectDictionary` provided method). Dispatch matches incoming frames against each server's configured rx COB-ID. COB-ID uniqueness (rx≠tx, no shadowing of the default `0x600/0x580 + node_id` or another server) is enforced in two layers: the macro rejects every statically-decidable collision at compile time (including node-relative bases vs the default), and `Node::new` has a real `assert!` (not debug-only) for the residual node-id-dependent case (an absolute COB-ID landing on the default for a particular node id). EDS export yes; EDS import pending (`sdo_server[N]` DSL only for now). Design: `_Tasks/additional-sdo-servers.md`; `vcan_node.rs` carries a diagnostics channel.
- All types use static allocation (heapless).
- `Dcf` (behind `alloc` feature) parses DCF/EDS files at runtime and applies configured values to local ODs or remote nodes via SDO.
- STM32 firmware uses Embassy channels to bridge async CAN driver and polling Node.

## OD Definition

Inline DSL (supports VAR, RECORD, ARRAY, and variable-length types):
```rust
object_dictionary! {
    #[export_eds(path = "./my_device.eds")]
    pub struct MyOd {
        [0x1000] device_type: u32 = 0x191, ro;
        [0x1008] device_name: visible_string<32>, ro;
        [0x1003] error_field: array<u32, 8>, ro;
        [0x6000] inputs: record {
            [1] input1: u8 = 0, ro, pdo;
        };
        [0x2020] firmware: domain<1024>, rw;

        tpdo[1](transmission_type = event_driven, inhibit_time = 50ms, event_timer = 1s) {
            input1,
        };
        rpdo[1](deadline = 500ms) {
            output1,
        };

        // Optional extra SDO server (0x1201) beside the implicit default 0x1200.
        sdo_server[2](cob_rx = node_id + 0x640, cob_tx = node_id + 0x5C0);
    }
}
```
PDO parameters accept CiA 301 keywords (`event_driven`, `event_driven_manufacturer`, `sync_acyclic`, `sync_cyclic(N)`, `rtr_sync`, `rtr_event`) or raw spec values (`transmission_type = 255`). Time parameters accept raw integers in spec wire units (inhibit_time in 100µs, event_timer/deadline in ms) or unit-suffixed literals: `50ms`, `500us`, `0.1s` (floats need a suffix; values must be exactly representable in the wire unit). Parameters are direction-checked at compile time: `event_timer`/`inhibit_time`/RTR types are TPDO-only, `deadline` is RPDO-only (`deadline` is the receive-side function of the spec's "event timer" field — same wire slot, comm param sub 5, exported to EDS as "Event Timer").

**Mapping mutability**: PDO mapping records are **immutable by default** — the mapping is a device invariant, SDO writes to `0x160N`/`0x1A0N` are rejected, and the EDS exports `AccessType=const`. Opt into CiA 301 dynamic mapping per PDO with `mapping = mutable` (exposes the standard unlock protocol: write 0 to sub 0, rewrite entries, restore the count — Pre-Operational only, exported as `rw`). Comm params (COB-ID, transmission type, deadline) stay `rw` either way. EDS import derives the kind from the mapping record's AccessType (`ro`/`const` → immutable, `rw` → mutable; a comm-only PDO without a mapping section imports as mutable so it can be mapped at runtime). Internally this is `MappingKind::{Immutable, Mutable}` on `PdoDef`.

**PDO mapping granularity and capacity**: mapping entries are `(index, subindex, bit_length)` only; CANopen has no source bit-offset field, so partial mappings always start at bit 0 of the mapped OD subobject. Hand-written DSL `bool` mappings use bit length 1; imported EDS mappings preserve their declared length, including 8-bit `BOOLEAN` mappings. Classic CANopen PDOs are limited to 64 payload bits. yacob's generated OD stores up to 64 mapping entries per PDO so bit-granular PDOs can use 64 one-bit mappings; static mappings and SDO remapping writes are rejected if the active total exceeds 64 bits. Immutable EDS exports list only active mapping entries. Mutable EDS exports list the full 64 writable entries and `sub0` includes `LowLimit=0`/`HighLimit=64`, because generic tools need implemented subindices to exist in the OD/EDS model before they can write them.

**Dummy PDO mappings** (CiA 301, static data type indices 0x0001–0x0007): mapping entries in that index range are padding — TPDOs emit zero bits, RPDOs skip the bits without OD writes or events. EDS import preserves dummy entries and re-export round-trips them (the CiA 306 `[DummyUsage]` section is ignored — import is lenient). Two deliberate extensions beyond the spec: dummies work on TPDOs too (CiA 301 defines dummy mapping receive-side only; we transmit zeros), and a dummy's bit length may differ from the static type's natural size (e.g. 0x0001 with 7 bits) so bit-granular padding composes with 1-bit bools. No DSL syntax yet — dummies come in via EDS import or runtime remapping only (see known-issues backlog).

PDO numbers go up to 512 (CiA 301 limit) and may be sparse (e.g. only `tpdo[1]` and `tpdo[5]`). PDOs 1-4 get the pre-defined COB-IDs (`0x180/0x280/0x380/0x480 + node_id` for TPDO, `0x200..0x500 + node_id` for RPDO) unless overridden; PDOs 5+ have no predefined COB-ID and require an explicit one (compile error otherwise): `cob_id = 0x1B1` (fixed) or `cob_id = node_id + 0x1B0` (node-relative — resolved with the actual node ID, so multiple devices can share one OD; exported to EDS as `$NODEID+0x1B0`, and `$NODEID` expressions in imported EDS round-trip as node-relative). Comm/mapping params always live at `0x1800/0x1A00 + number - 1` (TPDO) and `0x1400/0x1600 + number - 1` (RPDO); the engine slot ↔ OD index link is the `od_number` field on `TpdoConfig`/`RpdoConfig`. `Node::new` mirrors resolved COB-IDs back into the OD, so SDO reads of sub 1 return real values (never the internal 0 = "default pending") and PDO config survives `request_reset`. EDS import (`object_dictionary_from_eds!`) scans the full 512-PDO ranges. Note: `CanOpenEvent::decode` classifies PDO frames by the pre-defined COB-ID ranges only — PDOs with custom COB-IDs outside those ranges decode as the range they land in.

Generated besides the struct itself:
- `MyOd::EDS: &'static str` (uncompressed) and `MyOd::EDS_COMPRESSED: &'static [u8]` (deflate). Auto-registers 0x1021 (Store EDS, compressed) and 0x1022 (Store Format = 1) for device self-description via SDO.
- Address consts: `MyOd::INPUT1: (u16, u8)` per entry (usable as match patterns), `MyOd::ERROR_FIELD_INDEX: u16` for arrays.
- `MyOdChange` enum + `OdChanges` impl: one variant per writable entry (`Input1(u8)`, arrays as `Name(subindex, value)`, variable-length as unit variants), decoded by `node.next_change()`.
- `MyOdNode` type alias = `Node<MyOd, TPDO_COUNT, RPDO_COUNT, SDO_COUNT>` (`SDO_COUNT` = number of `sdo_server[N]` declarations, 0 if none).
- `PdoConfigSource` and `SdoServerConfigSource` impls for `NodeConfig::from_od`.

Variable-length types: `visible_string<N>`, `octet_string<N>`, `domain<N>` backed by heapless. Use `#[use_alloc]` attribute for `String`/`Vec<u8>` instead (no capacity needed).

24-bit types: `i24` (INTEGER24, 0x0010) and `u24` (UNSIGNED24, 0x0016) are 3 bytes on the wire (SDO and PDO — mapping bit length 24) but represented as `i32`/`u32` in Rust (sign-/zero-extended on write). Values outside the 24-bit range are silently truncated on read — the application must keep the field in range. `SdoDriver` has `read_i24`/`write_i24`/`read_u24`/`write_u24`; the EDS client codegen uses them.

Write validation via `#[validate_write(fn_name)]` — the SDO server calls the named method before committing writes:
```rust
object_dictionary! {
    #[validate_write(check_value)]
    pub struct MyOd {
        [0x2000] level: u16 = 0, rw;
    }
}
impl MyOd {
    fn check_value(&self, index: u16, sub: u8, data: &[u8]) -> Result<(), OdError> {
        // Return Err(OdError::ValueRange) to abort the SDO write
        Ok(())
    }
}
```

From EDS file (with per-entry capacity overrides and automatic PDO extraction):
```rust
object_dictionary_from_eds! {
    pub struct MyOd = "device.eds"
    with {
        [0x1008] capacity = 32,
        [0x1021] capacity = 4096,
    };
}
```

Typed async SDO client from EDS:
```rust
sdo_client_from_eds! {
    pub struct DeviceClient = "device.eds";
}
let client = DeviceClient::new(node_id);
let status: u16 = client.read_statusword(&mut can).await?;
client.write_controlword(0x0F, &mut can).await?;
```

Runtime DCF configuration (requires `alloc` feature):
```rust
let dcf = canopen_core::dcf::Dcf::parse(dcf_text)?;
dcf.apply(&mut od);                               // local OD
dcf.apply_remote(&sdo_driver, &mut can).await?;    // remote node via SDO
```

## Async SDO

`AsyncCan` trait for async CAN transports. `SdoDriver` drives complete SDO transfers:
```rust
let sdo = SdoDriver::new(target_node);
let mut demux = CanDemux::new(transport);
let mut port = demux.sdo_port(target_node);
let val = sdo.read_u32(0x1000, 0, &mut port).await?;
// Non-SDO frames buffered in demux:
while let Some(hb) = demux.try_recv_heartbeat() { ... }
```

## Features (canopen-core)

| Feature | Default | Effect |
|---|---|---|
| `alloc` | off | Enables `Dcf` runtime parser, `extern crate alloc` |
| `std` | off | Enables std error impls, implies `alloc` |
| `embassy` | off | Enables `OdEventSignal` (embassy_sync) and a real timer for `SdoDriver` timeouts on silent buses (embassy_time) |
| `defmt` | off | Enables defmt formatting |

## Transports

| Transport | Crate | Needs root | Use case |
|---|---|---|---|
| `SocketcanTransport` | canopen-linux | Yes (vcan setup) | Linux socketcan |
| `SlcanTransport<P: SerialPort>` | canopen-core | No | Serial SLCAN adapters |
| `UdpMulticastTransport` | canopen-linux | No | Cross-process testing, CI |
| `MailboxTransport` | canopen-core | N/A | ISR↔task bridging on MCU |

All implement `embedded_can::nb::Can<Frame = CanFrame>`.

The SLCAN protocol driver (`canopen_core::slcan`) is sans-IO and generic over
the `SerialPort` byte-stream trait, so it works in `no_std` (e.g. over an MCU
UART). Backends: `canopen_linux::slcan::UnixSerialPort` (unix-only, with
`slcan::open(path, bitrate)` handling the CDC-ACM DTR-reset init dance) and
`canopen_core::slcan::IoPort<T>` (`std` feature) wrapping any
`std::io::Read + Write`, e.g. the `serialport` crate on Windows/macOS.
